# Як створити зручний TypeScript full-stack monorepo

Вітаю, мене звати Герман. За ці роки я бачив, як багато команд створюють full-stack monorepo, доводять його до стану, коли він технічно працює, а потім постійно фіксять якісь проблеми, додають якісь хаки або просто забивають на потенційні покращення, тому що вони надто складні. І після достатньої кількості такого висновок часто простий: monorepo того не вартий.

Я не думаю, що проблема зазвичай у самому monorepo. Частіше проблема в налаштуванні, яке одного разу змусили працювати, але так і не зробили комфортним для щоденної роботи. У цій статті я хочу показати підхід, який я використовую, щоб full-stack monorepo залишався плавним у роботі, практичним і близьким до типової розробки застосунків.

Я пишу це для інженерів, яким потрібен один репозиторій для client, API та shared TypeScript-коду, але які не хочуть, щоб робочий процес із кожним місяцем ставав важчим. Як приклад я використаю власний репозиторій не як універсальний шаблон, а як конкретну демонстрацію рішень, завдяки яким monorepo залишається для мене зручним.

## Чому я обираю monorepo для full-stack розробки?

Я не думаю, що кожен full-stack застосунок має жити в monorepo. Якщо API і client можуть повноцінно бути реалізовані в межах одного framework по типу Astro або якщо ці частини майже не залежать одна від одної, monorepo може бути зайвим.

Але я часто будую системи, у яких API і client тісно пов'язані, хоча при цьому мають залишатися окремими застосунками. Мені може бути потрібен окремий API зі своєю моделлю виконання, власним deployment, WebSockets, job queue, які складно реалізувати в межах full-stack framework. У таких випадках monorepo стає хорошою золотою серединою.

Він тримає застосунки достатньо близько, щоб ділити між ними контракти та runtime-код, але не змушує запаковувати все в одну структуру, продиктовану конкретним full-stack framework. У цьому репозиторії API це застосунок на Hono, а client це React-застосунок на Vite. Client може імпортувати тип застосунку API, щоб побудувати type-safe запити до API, а якийсь shared код може існувати в shared бібліотеці.

Альтернатива часто виявляється дорожчою, ніж здається спочатку. Щойно я розношу тісно пов'язану систему по окремих репозиторіях, мені зазвичай потрібні приватна публікація пакетів, координація версій і складніший CI. З мого досвіду це часто створює більше проблем, ніж monorepo, а не менше.

## Чого я хочу від full-stack monorepo?

Перш ніж обирати інструменти, я намагаюся визначити, що саме хочу отримати від monorepo. Якщо цього не зробити, надто легко почати збирати features просто тому, що інструменти для monorepos їх пропонують.

Для full-stack TypeScript monorepo мої базові вимоги доволі прості:

- Чіткі межі між застосунками, які можна запускати, спільним runtime-кодом і tooling.
- Можливість запускати scripts з кореня репозиторію, коли це зручно, і водночас залишати кожен workspace повноцінним сам по собі.
- Спільний код без публікації внутрішніх пакетів і без rebuild після кожної дрібної зміни.
- Dev scripts, які природно реагують на зміни в залежностях.
- CI, який запускає перевірки лише там, де зміни справді мають значення.
- Docker-образи, у яких є лише те, що потрібно цільовому застосунку.
- Git-історія та конвенції ведення репозиторію, які залишаються зрозумілими для звичайних розробників застосунків.

Цей список важливий, бо він змінює те, як я оцінюю інструменти. Я не шукаю найфункціональніший monorepo setup. Я шукаю setup, яке вирішує ці практичні проблеми з необхідним мінімумом.

## Прості межі між workspaces

Перше, чого я хочу від monorepo, це структура, яка має сенс відразу після відкриття, без жодних пояснень. Я не хочу верхньорівневих папок, чий сенс змінюється від проєкту до проєкту. Я хочу межі, які підказують, що переді мною, ще до того, як я відкрию код.

Саме тому я використовую дуже простий поділ:

```text
apps/
├── api
└── client

libs/
└── shared

packages/
├── eslint-config
├── prettier-config
└── tsconfig
```

`apps/` містить застосунки, які можна запускати. `libs/` містить спільний runtime-код. `packages/` містить tooling і спільну конфігурацію. Це не революційна структура, але саме тому вона мені й подобається. Вона добре масштабується, її легко сканувати очима, і вона тримає runtime-код окремо від tooling-задач.

На рівні пакетного менеджера я теж тримаю основу максимально простою. npm workspaces уже дають мені все, що тут потрібно: dependency resolution між workspaces, symlinks на внутрішні пакети через `node_modules` і запуск scripts з кореня.

```json
{
  "name": "@example/root",
  "private": true,
  "workspaces": ["apps/*", "libs/*", "packages/*"]
}
```

Я знаю, що багато команд віддають перевагу pnpm або Yarn, і це чудові інструменти. Це не аргумент про те, що npm універсально кращий. Для цього репозиторію npm уже достатньо.

## Buildless TypeScript internal packages

Найбільший вибір у цьому налаштуванні стосується того, як я працюю з внутрішніми пакетами. Багато розмов про monorepos одразу виходять із припущення, що для них спочатку треба робити build, а вже потім споживати їх як скомпільований результат. У певних середовищах це може бути правильним вибором, але за замовчуванням я цього не хочу.

Для внутрішнього коду, який живе і помирає в межах одного репозиторію, я віддаю перевагу тому, що Turborepo в документації називає [Just-in-Time Internal Package](https://turborepo.dev/docs/core-concepts/internal-packages#just-in-time-packages). На практиці це означає, що пакет вказує прямо на TypeScript-джерело, а решта toolchain споживає його без окремого build кроку.

```json
{
  "name": "@example/shared",
  "private": true,
  "type": "module",
  "main": "src/index.ts",
  "imports": {
    "#src/*.ts": "./src/*.ts"
  }
}
```

Одне це рішення прибирає напрочуд багато інфраструктурного коду, який зазвичай з'являється в monorepo. Водночас воно забороняє використовувати TypeScript aliases, бо для них уже потрібен build крок.

Якби я захотів aliases для внутрішніх пакетів, мені довелося б додати build у `dist/`, post-build rewrite через щось на кшталт `tsc-alias`, а потім ще окремо підтримувати watch-цикл для rebuilds. Через `turbo watch` це означало б або зайві рестарти dev process, або паралельний цикл rebuild залежностей на кожній зміні. Якщо ж захотіти тоншої поведінки, де перебудовуються лише monorepo залежності, а локальні зміни й далі обробляє watcher самого застосунку, це вже легко штовхає до custom watch script поверх чогось на кшталт `turbowatch` і навіть окремого внутрішнього scripts package.

До цього часто додається ще й редактор: після кількох rebuilds сервер TypeScript інколи перестає коректно синхронізуватися, тож доводиться або перезапускати його вручну, або підпирати setup `references` без `composite`, щоб IDE залишалася в курсі актуального стану. Такий компроміс може бути виправданим, якщо пакет треба публікувати або toolchain справді вимагає скомпільований результат, але без реальної потреби я б у нього не заходив.

## Чому розробка залишається плавною?

Рішення з пакетами без окремого build працює, тому що вибір runtime і tooling його підтримує, а не бореться з ним.

Цей репозиторій орієнтується на сучасний Node.js з `erasable syntax` і спирається на прямий запуск TypeScript entrypoints. Пакет API використовує `node src/index.ts` для старту і `node --watch src/index.ts` для розробки. Shared бібліотека теж працює напряму з вихідним кодом. На практиці це означає, що я можу змінити код у `@example/shared`, природно перезапустити чи повторно виконати все через звичайний tooling і уникнути звичного циклу "змінив код, зробив rebuild пакета, перезапустив застосунок, оновив стан редактора".

Конфігурація TypeScript тут навмисно підігнана під цю модель:

```json
{
  "compilerOptions": {
    "target": "esnext",
    "verbatimModuleSyntax": true,
    "strict": true,
    "erasableSyntaxOnly": true,
    "allowImportingTsExtensions": true,
    "rewriteRelativeImportExtensions": true,
    "module": "NodeNext",
    "noEmit": true
  }
}
```

Node може запускати TypeScript entrypoint API-застосунку, який імпортує внутрішні пакети без окремого build, без додаткових прошарків. Vite і Vitest можуть робити те саме під час client-розробки, тож мені не потрібен окремий monorepo режим розробки, нашарований поверх звичайної розробки.

Тут також є full-stack перевага з імпорту API-типів:

```ts
// apps/api/src/app.ts
export const app = new Hono().get('/ping', (c) => c.text(ping()));
export type App = typeof app;

// apps/client/src/api.ts
import { type App } from '@example/api/app';
import { hc } from 'hono/client';

const api = hc<App>('http://localhost:3000');

// somewhere in the client code
const res = await api.ping.$get();
console.log(await res.text());
```

Я експортую тип застосунку із серверного пакета і споживаю його на client через `hc<App>()`. Якщо я зламаю серверний контракт несумісною зміною, client може впасти на етапі перевірки типів замість того, щоб ця невідповідність дожила до runtime. Коли щось не можна або не варто експортувати прямо із серверного пакета, я виношу це в shared бібліотеку, і контракт між застосунками залишається поруч із кодом, який його використовує.

## Конвенції, які роблять щоденну роботу плавнішою

Ці конвенції стосуються не лише механіки виконання. Вони також роблять щоденну роботу плавнішою, бо зменшують кількість рішень, які інженерам доводиться приймати щодня.

Я тримаю спільні configs ESLint, Prettier і TypeScript у `packages/eslint-config`, `packages/prettier-config` і `packages/tsconfig` та ставлюся до них як до звичайних пакетів workspaces.

Для спільного Prettier: кожен workspace додає `@example/prettier-config` і вказує на нього у своєму полі `prettier` у package.json. `.prettierignore` неможливо поділити таким самим способом, тому я тримаю один у корені й створюю його в кожному workspace.

```json5
// packages/prettier-config/src/index.json
{
  "semi": true,
  "trailingComma": "all",
  "singleQuote": true,
  "printWidth": 120
}

// apps/api/package.json
{
  "devDependencies": {
     "@example/prettier-config": "*"
  },
  "prettier": "@example/prettier-config"
}
```

Для ESLint я зазвичай хочу, щоб спільний ESLint-пакет віддавав кілька очевидних базових конфігів на кшталт `base`, `node` або `client`, а кожен workspace при цьому тримав невеликий локальний `eslint.config.mjs`. У цьому repo API і shared бібліотека можуть просто експортувати `eslintConfig.node`, тоді як client-файл здебільшого містить client-шар поверх `eslintConfig.base`, а саме правила для браузера, React Hooks і Vite.

```js
// packages/eslint-config/src/index.mjs
export const base = defineConfig(
  eslint.configs.recommended,
  ...tseslint.configs.recommended,
  eslintPluginPrettierRecommended,
);

// apps/client/eslint.config.mjs
export default defineConfig([...eslintConfig.base, reactHooks.configs.flat.recommended, reactRefresh.configs.vite]);
```

TypeScript іде тим самим шляхом, але форма пакета простіша. Спільний `tsconfig`-пакет зазвичай просто тримає файли на кшталт `node.json` у корені пакета, а потім кожен workspace розширює те, що йому потрібно. У цьому repo API і shared бібліотека розширюють `@example/tsconfig/node.json`, а client тримає власні `tsconfig` файли, орієнтовані на Vite, бо Vite має свої обмеження.

```json5
// packages/tsconfig/node.json
{
  "compilerOptions": {
    "target": "esnext",
    "verbatimModuleSyntax": true,
    "erasableSyntaxOnly": true,
    "module": "NodeNext",
    "noEmit": true
  }
}

// apps/api/tsconfig.json
{
  "extends": "@example/tsconfig/node.json"
}
```

Схожу логіку я застосовую і для commits. `Husky` і `lint-staged` проганяють виправлення перед commit, а найближчий config обробляє staged files, тож файли в root можуть залишатися на root правилах, тоді як застосунки й бібліотеки тримають власний локальний setup. При цьому `apps`, `packages` і `libs` ігноруються для root перевірок, щоб форматування, лінтинг і перевірка типів у root були сфокусовані на файлах, які належать йому.

```bash
# .husky/pre-commit
npx --no-install lint-staged
```

```json5
// package.json
"lint-staged": {
  "*.{js,cjs,mjs,json,yml,md}": "prettier --write",
  "*.ts": "eslint --fix"
}

// apps/client/package.json
"lint-staged": {
  "*.{js,cjs,mjs,json,yml,md,html,css}": "prettier --write",
  "*.{ts,tsx}": "eslint --fix"
}
```

Conventional commits також допомагають. У monorepo scope особливо корисний. Commit на кшталт `feat(api):` або `fix(client):` підказує мені, яка саме частина системи змінилася, ще до того, як я відкрию diff, тоді як звичайний `feat:` зазвичай означає, що зміна зачіпає кілька застосунків або весь репозиторій. Це також допомагає, коли мені потрібно згенерувати changelog. У цьому repo і `update:changelog`, і release flow в `scripts/release.ts` виграють від передбачуваних типів і commit scopes. Це маленька конвенція, яку покривають `commitlint` і `Husky`, але з часом вона окупається.

```bash
# .husky/commit-msg
npx --no-install -- commitlint --edit "$1"
```

```json
// .commitlintrc.json
{
  "extends": ["@commitlint/config-conventional"]
}
```

Також мені подобаються custom теги PR на кшталт `shared`, `api` або `client`, бо вони дозволяють фільтрувати PRs й розуміти, що саме було зачеплено, ще до читання файлів.

А щодо версіонування, то я тримаю версіонування простим. У цьому репозиторії використовується одна версія для root пакета і всіх workspaces. Тут немає потреби в окремих версіях для кожного пакета і, відповідно, немає потреби у складному процесі оновлення версій.

## Де Turborepo справді приносить користь

Turborepo не є обовʼязковим для цього setup, але в певних місцях він справді корисний.

Найочевидніший приклад це affected CI. Я хочу, щоб репозиторій розумів зв'язки між пакетами і запускав перевірки лише там, де зміна справді має значення, а Turborepo вже досить добре вміє це робити.

```yaml
# .github/workflows/affected.yml
env:
  # https://github.com/vercel/turborepo/issues/9320
  TURBO_SCM_BASE: ${{ github.event_name == 'pull_request' && github.event.pull_request.base.sha || github.event.before }}

steps:
  - run: npx turbo run format --affected
  - run: npx turbo run lint --affected
  - run: npx turbo run typecheck --affected
  - run: npx turbo run test --affected
```

Мені також потрібні окремі задачі в `turbo.json` для перевірок, які мають реагувати на файли в корені на кшталт `README.md`:

```json5
// turbo.json
"//#format": {
  "cache": false,
  "inputs": ["$TURBO_DEFAULT$", "!apps/**", "!libs/**", "!packages/**"]
}
```

У цьому репозиторії я явно задаю `TURBO_SCM_BASE` у GitHub Actions, щоб допомогти Turborepo знайти правильну ціль під час використання `--affected`, а root tasks у `turbo.json` потрібні для того, щоб у такі запуски потрапляли ще й файли з root, а не лише зміни у workspaces.

Docker це інший очевидний приклад. `turbo prune` дозволяє мені будувати образ лише з того коду і залежностей, які потрібні цільовому застосунку, замість того щоб тягнути весь репозиторій у контекст збірки. У цьому repo Dockerfile для API використовує `turbo prune --scope=@example/api --docker` саме з цієї причини. Це реальна цінність, а не просто абстракція заради самої абстракції.

Саме тому я тут не використовую Nx. Я вважаю, що він добре працює, коли репозиторій залишається всередині його моделі, але разом із цим приходить більше абстракції і більше магії.

## Додаткові речі, які роблять життя приємнішим

Коли основний workflow вже стабільний, кілька менших рішень справді роблять життя з цим репозиторієм приємнішим.

Одне з них це генерація коду. У monorepo багато повторюваної роботи: створити пакет, додати scripts, під'єднати спільні configs, заповнити базову структуру і переконатися, що жодна дрібниця не загубилася. У цьому repo я використовую для цього Plop і показую приклад його використання у root `generate:package` скрипті, але той самий підхід працює всюди, де структура повторюється, наприклад якщо йдеться про створення нового microservice разом з Terraform schema змінами. Це не центральна архітектурна частина, але воно рятує мене від нудних помилок через копіювання.

Ще одна така річ це те, як я працюю з AI-агентами у репозиторії. У monorepo я віддаю перевагу запуску агента з monorepo root. Це тримає стан агента, дозволи і пам'ять в одному місці, а не розмазує їх по різних workspaces. Водночас, коли агенту потрібно працювати всередині вкладеного застосунку чи бібліотеки, він може автоматично завантажити локальний файл `AGENTS.md` / `CLAUDE.md`, тож я все одно можу прикріплювати сфокусовані інструкції до конкретних частин repo, коли це потрібно. Для репозиторію з кількома застосунками і спільними пакетами така модель запуску з кореня відчувається набагато чистішою.

## Висновок

Для мене зручний TypeScript monorepo це не про те, щоб зібрати найпросунутіший stack. Це про те, щоб прибрати зайві шари, доки репозиторій знову не почне відчуватися як типова розробка застосунків. npm workspaces відповідають за локальне зв'язування пакетів, внутрішні пакети без окремого build прибирають нескінченний цикл rebuilds, сучасний Node.js робить TypeScript workflows прямолінійними, а Turborepo залишається в тих місцях, де він дійсно приносить користь.

Я не подаю цей репозиторій як ідеальний шаблон, який має копіювати кожна команда. Це демонстрація ідеї та набору компромісів. Але якщо ви будуєте full-stack TypeScript проєкт і втомилися від monorepos, які здаються важчими за сам продукт, то саме з цього напрямку я б починав.
